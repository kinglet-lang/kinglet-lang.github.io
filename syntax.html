Language & toolchain.

Kinglet is a C++-like surface with a smaller rule set: value types, monomorphized generics, postfix match, nullable T?, and C++20-style modules. This page walks through the project workflow, the kinglet.nest manifest, the CLI, and the parts of the language that differ most from everyday C++.

Quick start

After installing kinglet from the homepage, you can compile a single file immediately or scaffold a project. Projects are driven by a kinglet.nest manifest at the root.

  1. One-off file — compile and run without a manifest: kinglet hello.kl
  2. New project — run kinglet init and answer two prompts (project name and parent directory), or pass a path: kinglet init myapp
  3. Build — compiles the default target to .kinglet/out/<target>: kinglet build
  4. Run — executes the built binary (pass args after run): kinglet run
kinglet init
Create a new Kinglet project

?  Project name (kinglet-app) › demo
?  Location (.) › ~/Code

✓  Created project demo

   ~/Code/demo/
   kinglet.nest
   src/main.kl

   Next steps
     cd ~/Code/demo
     kinglet build

Press Enter to accept the defaults (kinglet-app in the current directory). The manifest’s project name is taken from the final directory name. kinglet init myapp skips the prompts and creates ./myapp/ directly. In a non-interactive shell with no path, it defaults to kinglet-app.

after build
kinglet build
kinglet run
./.kinglet/out/app          # same binary, by path

Native linking needs clang++ on your PATH.

kinglet.nest

Every multi-file project has a kinglet.nest at its root. The manifest declares build targets (what to compile and link) and formatter settings. Module names live in .kl source files themselves — the manifest does not map names to paths.

What init creates

kinglet.nest
project "demo" version "0.1.0"

target app {
  kind    = "binary"
  sources = ["src/"]
}

build {
  default = "app"
  cache   = ".kinglet/cache"
  out     = ".kinglet/out"
}
src/main.kl
export module app;
using io;

int main() {
  io::out.line("Hello from demo!");
  return 0;
}

Full manifest

A larger project can define multiple targets, directory globs, and dependencies:

kinglet.nest
project "myapp" version "0.1.0"

target calc {
  kind    = "binary"
  sources = [
    "src/calc.kl",
    "math/",
    "geo/",
    "utils/config.kl",
  ]
}

target app {
  kind    = "binary"
  sources = ["src/app.kl", "math/"]
}

build {
  default = "calc"
  out     = ".kinglet/out"
  cache   = ".kinglet/cache"
}

fmt {
  indent    = 2
  max_width = 80
}
  • project — required line: project "name" version "x.y.z"
  • target — kind is binary (default, also spelled executable), library, test, or object; sources lists .kl files and/or directories (trailing / globs every *.kl inside); deps lists other targets whose modules become importable
  • build — default is built by bare kinglet build; out and cache override .kinglet/out and .kinglet/cache
  • fmt — read by kinglet fmt: indent, max_width, newline ("lf"/"crlf"), trailing_comma, extensions

A binary target must have exactly one source file with int main(); the compiler finds the entry automatically. The output executable is named after the target: .kinglet/out/calc.

CLI

The release ships one kinglet binary. klet is a hard link to the same file — klet build and kinglet build are identical.

  • ▸ kinglet <file.kl> Compile and run a single source file (no manifest needed).
  • ▸ kinglet init [path] Scaffold kinglet.nest and src/main.kl. Interactive: asks for project name and parent directory. With a path, creates that directory directly.
  • ▸ kinglet build [--quiet] [target] Build a target from the nearest kinglet.nest. Without a name, builds build.default. --quiet suppresses progress output.
  • ▸ kinglet run [<file.kl> | args…] Run .kinglet/out/<default-target> with optional arguments. kinglet run file.kl compiles and runs that file instead.
  • ▸ kinglet fmt [--check|--write] [paths…] Format .kl sources using fmt { … } from the manifest. --check exits non-zero if formatting would change files (for CI); --write is the default when neither flag is given. --stdin formats stdin instead of paths; --config <path> overrides which manifest to read.
  • ▸ kinglet prune [--dry-run|-n] [--all] [root] Remove stale object files from .kinglet/cache whose sources no longer exist. --dry-run/-n reports without deleting; an optional path targets a project root other than the current directory.

Compiler flags

When invoking kinglet on a .kl file directly, these modes apply:

  • --check — type-check only, no codegen
  • --tokens / --ast — dump lexer or parser output
  • --ir — print KIR (middle-end IR)
  • --native <out> / -o <out> — write a native executable (requires LLVM build)
  • -g — emit DWARF debug info alongside --native/-o
  • --obj-cache <dir> — cache object files for incremental native builds

Modules

Kinglet uses C++20-modules-style identity: each file declares export module name; and other files import by logical name, not file path. The compiler resolves imports against the current target's sources plus deps.

math/basic.kl
export module math.basic;

pub int square(int x) { return x * x; }
src/app.kl
export module app;
using io;
import math;            // group: every module named math.*
import geo.point;       // exact: just geo.point

int main() {
  io::out.line("{}", math::basic::square(4));
  return 0;
}
  • Exact import — import math.basic; loads the file that exports module math.basic
  • Group import — import math; pulls every module whose name starts with math.; call via math::basic::square
  • Runtime namespaces — io, fs, sys, rt are built in; open with using io;, never import
  • Using — using namespace math.basic; or using alias = math.basic;

Listing "math/" in sources only adds .kl files from that directory — it does not impose a namespace. Each file still declares its own export module.

Language

Most everyday syntax is familiar C++: braces, semicolons, for/while, if/else. There is no ++/-- (write i = i + 1), no switch (use match), and when/spawn/select are reserved but not implemented yet.

Types

Primitives: int, sized integers (int32, uint64, …), float/float32/float64, bool, string, void, byte, char. Suffix forms: int[], {string: int} maps, T? nullables, &T / &mut T references, Box<int> generics.

Functions

int add(int a, int b) { return a + b; }
T identity<T>(T x) => x;              // expression body

Generic functions monomorphize at compile time. The entry point for binaries is always int main().

Control flow & declarations

guard is an early-exit check whose else block must not fall through. Destructuring pulls array elements into named bindings, with ...rest capturing everything left over. Comparisons chain mathematically: a < b < c desugars to a < b && b < c, evaluating the middle operand once. 

guard b != 0 else { return -1; }

auto [a, b, ...rest] = nums;   // array/tuple unpacking

if (0 <= i < n) { ... }        // 0 <= i && i < n

Structs & enums

struct Point { int x; int y; }
Point p { 3, 4 };                     // struct literal

enum Shape {
  Circle(float),
  Rect(float, float),
  None,
}

Enum variants can carry typed payloads. Trailing commas are allowed. Use pub on declarations you want visible outside the module.

Pattern matching

match is postfix: value match { arms }. Arms are pattern => expr, comma-separated. The checker enforces exhaustiveness on enum and nullable subjects. 

return s match {
  Shape::Circle(let r)     => 3.14 * r * r,
  Shape::Rect(let w, let h) => w * h,
  Shape::None              => 0.0,
};

Patterns include _, let x, literals, enum destructuring, array [let a, let b], and struct Point { let x, let y }. Guards: let x if (x >= 90) => ….

Concepts

Concepts constrain generic code (like traits/interfaces). At least one type parameter is required; method signatures need explicit return types. 

concept Printable<T> {
  string to_string(T value);
}

string s = n.to_string();   // uniform function call syntax

f(x) and x.f() desugar to the same call when one form fits. The pipe operator x |> f passes x as the first argument.

Errors & nullables

T? is a nullable type. Postfix ? unwraps or propagates failure; ?: is Elvis/coalesce; try/catch handles errors in a block. 

int? a = int("123") ?: -1;
int n = int("bad")?;                  // in T? function: return null on failure

int? x = int(text) ?: let err => 0;   // binds the CastError as err

try {
  n = int("bad")?;
} catch (let ex: CastError) {
  n = -99;
}

The binding form of ?: requires a fallible cast on the left (like int(...)) and keeps the result nullable, so the target must be T?.

Built-ins & I/O

string, array, and map have methods like len(), push, contains — no import needed. After using io;: 

io::out("{} {}\n", a, b);
io::out.line("x={}", v);
string line = io::in();
